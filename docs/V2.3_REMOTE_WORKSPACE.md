# MOGU 2.3 — Remote Workspace

```yaml
status: IMPLEMENTED · DEFAULT-OFF
branch: capability/2.3
baseline: capability/2.2 (v2.2.0-personal-stable)
```

## Objective

Add a **remote message-driven Task Source** so Telegram / QQ / WeChat (and future channels)
can submit work into the existing MOGU pipeline without rewriting Agent Runtime, Trust Plane,
PermissionProxy, TaskStore schema, Brain Adapter, or Coding Worker.

```text
Remote Message
  → TaskRequest
  → PermissionProxy (via RemotePermission + gateCommand)
  → RemoteTaskQueue (TaskStore-backed facade)
  → Brain / SkillRuntime
  → TaskResult
  → RemoteOutbox / channel adapter
```

## Non-goals

- No DAG / cross-skill self-healing / global scheduler (still later work)
- No Trust Plane redesign
- Adapters must not call Brain, Skill handlers, or Runtime directly

## Settings

```json
{
  "remote": {
    "enabled": false,
    "telegram": false,
    "qq": false,
    "wechat": false,
    "requireApproval": true,
    "allowAutoExecute": false
  }
}
```

All channel switches default **OFF**. Elevated capabilities (`WRITE` / `DELETE` / `RUN` / `SYSTEM`)
require admin `YES`/`NO` when `requireApproval` is true.

## Commands

`/start` `/help` `/status` `/task` `/cancel` `/retry` `/log`

## Hard constraints

- Remote must not bypass PermissionProxy, TaskQueue facade, Audit, or Trust Plane
- TaskStore schema unchanged (`source` remains an existing allowlist value; remote marks `kind: "remote"`)
- Gateway-accepted OpenClaw tasks are untouched by this layer’s cancel path except via existing abort hooks when wired
