## Positioning

```text
MOGU AI 2.3 Remote Workspace
= 已实现能力
≠ 已开启能力
```

Remote is a **Task Source**, not a new Runtime. It turns inbound channel messages into the existing:

```text
Remote Message → Task → Permission → Brain/Skill → Result → Channel
```

## Architecture

```text
Adapters (Telegram / QQ / WeChat)     transport only
        ↓
RemoteInbox / RemoteSession
        ↓
RemoteGateway.submitTask             sole business entry
        ↓
RemotePermission → PermissionProxy
        ↓
RemoteTaskQueue → TaskStore          kind: remote
        ↓
SkillRuntime | AgentRunService
        ↓
RemoteOutbox / NotificationService
        ↓
Adapter.send
```

Details: `docs/V2.3_REMOTE_WORKSPACE.md`  
Security audit: `docs/V2.3_SECURITY_REVIEW.md`

## Trust boundary

| Must not bypass | Status |
|-----------------|--------|
| PermissionProxy | Enforced in RemotePermission via `gateCommand` |
| TaskQueue facade | Sole execute path after authorize |
| Audit / desktop confirm for L2/L3 | Inherited from PermissionProxy |
| Adapter → Skill/Brain direct calls | Absent (static audit PASS) |
| Remote → shell | Absent (no child_process in remote/) |

Elevated remote capabilities require admin **YES/NO** when `requireApproval=true` (default).

## Known limitations

1. Live phone validation and 7-day stability are owner-operated, not CI-fabricated.
2. READ tasks may complete via SkillRuntime (`mogu.memory`) before a full Brain loop depending on wiring.
3. 30-minute away-from-desk soak is not automated here; completion notification shape is verified in simulation.
4. IPC `remote:inject|submit|approve` remains a privileged desktop surface — keep remote disabled when unused.
5. TaskStore schema unchanged (`source` allowlist); remote identity is `kind: "remote"`.
