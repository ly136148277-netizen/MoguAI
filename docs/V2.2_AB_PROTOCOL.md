# MOGU 2.2 Neural Layer — A/B Protocol

## Purpose

This protocol measures the attributable effect of the complete single-task Neural Layer while
protecting the 2.1 baseline. It does not evaluate 2.3 DAG scheduling, cross-skill self-healing,
global scheduling, or migration.

## Preregistered comparison

Each baseline/treatment pair must use the same:

- task IDs and order
- repository commit and dirty-tree policy
- exact provider and model IDs available to both arms
- prompts, sampling settings, tool versions, hardware class, and network policy
- maximum wall time, input/output tokens, tool calls, steps, repair iterations, and monetary budget
- permission decisions and writable-root policy

Baseline has every 2.2 flag off and exercises the unchanged 2.1 behavior. Treatment enables only the
preregistered 2.2 flags. It may use configured model routing only when the compared arms have an
explicit equal-budget model policy for the stated experiment.

`allowModelFallback` defaults to `false`. A treatment may enable it only through an explicit,
preregistered user opt-in with an ordered allowed-profile policy; otherwise an unavailable selected
model is `BLOCKED`. Unrecorded model substitution invalidates the pair.

## Task sets and contamination

Use exposed development tasks and private synthetic or owner-approved real-repository fixtures while
implementing 2.2. Record every exposed task ID. Development outcomes may tune the Neural Layer but
are not holdout evidence.

The MOGU 2.1 sealed holdout at `benchmarks/swe-bench/holdout/manifest.json` remains frozen and
unopened. It is not a 2.2 development set and must not be modified, sampled, or used for tuning.
Any future 2.2 holdout requires a separately approved manifest and preregistration.

## Required run record

Every arm writes immutable, secret-free evidence containing:

- code commit, dirty-tree state, task identity, protocol version, and configuration hash
- enabled flags and sanitized model-profile/task-policy identifiers
- selected model ID and route reason, or typed `BLOCKED` reason
- initial plan, bounded revisions, context allocations, tool sequence, and verification outcomes
- consumed tokens, tool calls, steps, repair iterations, wall time, and estimated cost
- terminal result and stop reason

Decision traces must not contain prompts or tool payload fragments that can expose credentials.
Configuration hashes exclude secret values.

## Metrics

- **Primary:** verified task completion rate and paired resolved outcomes.
- **Routing:** valid-route rate, typed-block rate, and zero silent fallback.
- **Planning:** plan completion, revision count, and unsupported-step rate.
- **Context:** target-evidence retention, truncation rate, and token use.
- **Tools:** successful authorized tool transitions, tool errors, and duplicate side effects.
- **Closed loop:** verification rate, repair success, exhausted-budget rate, and false success rate.
- **Efficiency:** wall time, model tokens, tool calls, steps, repair iterations, and estimated cost.
- **Safety:** permission bypass, path escape, secret persistence/exposure, unapproved write, duplicate
  external submission, or silent fallback. Every safety count must be zero.

## Promotion gates

Run at least two reproducible development comparisons with fixed seeds before requesting holdout or
default-on review. A candidate must:

1. show no regression in verified completion;
2. improve its preregistered target metric or efficiency measure;
3. remain within every configured per-task budget;
4. produce complete, secret-free decision traces; and
5. record zero safety violations and zero silent model fallbacks.

Results must include paired task outcomes and uncertainty; a small set cannot support a claim of
general superiority. Failed or blocked runs are reported, never silently removed.

## Decisions

The decision package recommends one of `reject`, `keep-default-off`, or `candidate-default-on` for
each flag and for the integrated Neural Layer. Default-on requires owner review after regression,
security, packaged Windows, and protocol evidence. No 2.2 result promotes a 2.3 capability.
