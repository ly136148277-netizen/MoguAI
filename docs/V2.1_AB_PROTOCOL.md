# MOGU 2.1 GPT-5.6 A/B and Holdout Protocol

## Preregistered comparison

The baseline and treatment must use the exact same task set and:

- provider and exact GPT-5.6 model ID
- system/task prompt inputs
- temperature and sampling parameters
- maximum steps, tokens, wall time, and monetary budget
- repository commit, tool versions, hardware class, and network policy

Treatment changes only the named 2.1 feature flags. A provider/model substitution invalidates the
pair and records `BLOCKED`, not a fallback score.

## Development sets

Use previously exposed SWE-bench Lite tasks and private synthetic fixtures for implementation and
debugging. Run each comparison twice with fixed seeds. Development results may tune behavior but
cannot be reported as holdout evidence.

## Sealed holdout

`benchmarks/swe-bench/holdout/manifest.json` is selected from SWE-bench Verified after excluding
instance IDs present in historical prediction files and committed samples. It contains only task
identity bindings, never problem statements, tests, hints, or gold patches.

Before a holdout run:

1. Record branch commit, dirty-tree status, model configuration hash, feature flags, and budgets.
2. Verify every task binding against the fetched source dataset.
3. Do not inspect individual outcomes until all paired runs finish.
4. Do not tune on holdout results. A failed preregistered threshold returns the capability to
   development with a new future holdout.

## Metrics

- Primary: resolved rate and verified completion rate.
- Retrieval: top-k target-file recall and definition/reference precision.
- Verification: test-discovery precision/recall and closed-loop verification rate.
- Efficiency: wall time, model tokens, tool calls, and estimated cost.
- Reliability: crash, retry, checkpoint recovery, and deterministic cleanup rates.
- Safety: permission violations, path escapes, secret exposure, unapproved writes, or silent model
  fallback. Every safety count must be zero.

## Promotion thresholds

A treatment is eligible for holdout only when two consecutive development runs satisfy:

- no primary-metric regression
- at least 10 percentage points improvement in its targeted capability metric, or at least 15%
  relative efficiency gain at equal resolution
- zero safety violations
- p95 wall time no worse than 25% unless resolution improves by at least 10 percentage points

Holdout eligibility requires the same safety rule and a positive primary effect. Because 20 tasks
have wide uncertainty, confidence intervals and per-task paired outcomes must be included; no claim
of general superiority may rely on this holdout alone.

## Output

Each run writes immutable JSON with task IDs, configuration hashes, aggregate metrics, paired
outcomes, failures, and artifact hashes. The final decision package adds License/security status,
maintenance cost, and a recommendation of `reject`, `keep-default-off`, or `candidate-default-on`.
