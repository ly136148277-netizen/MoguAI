## Audit questions

| # | Question | Result | Evidence |
|---|----------|--------|----------|
| 1 | Do all Remote Adapters enter via RemoteGateway / TaskSource? | **PASS** | `RemoteManager._registerAdapter` only binds `adapter.on("message") → taskSource.ingest → gateway.handleInbound`. Adapters expose `receive/send/upload/download` only. |
| 2 | Can Remote bypass PermissionProxy? | **PASS (no bypass found)** | `RemoteGateway.submitTask` always calls `RemotePermission.authorize` → `gateCommand(permissionProxy, …)` before `RemoteTaskQueue.enqueue`. Denied auth never enqueues. |
| 3 | Can an Adapter call Skill handlers directly? | **PASS** | Adapter modules require only `BaseAdapter` + `RemoteTypes`. No `skills/`, `handlers/`, `SkillRuntime`, or Brain imports. |
| 4 | Can a remote message execute shell directly? | **PASS** | No `child_process`, `exec`, `spawn`, or terminal APIs under `src/main/remote/**`. Shell remains behind existing PC/Coding skills + PermissionProxy. |
| 5 | Can Remote bypass TaskQueue? | **PASS** | Execution path is `gateway.submitTask → permission.authorize → queue.enqueue → execute hook`. No alternate execute path from adapters. |

## Default-Off and privilege

- `settings.remote.enabled|telegram|qq|wechat` default `false` (exact `true` opt-in).
- `allowAutoExecute` default `false`.
- `requireApproval` default `true`.
- Elevated capabilities (`WRITE` / `DELETE` / `RUN` / `SYSTEM`) require admin `YES` in addition to PermissionProxy when `requireApproval` is true.
- Approval timeout fail-closed → `NO`.
- Tokens resolved via SecretStore id `telegramBotToken`; never stored in `settings.json` sanitizer path for remote flags.

## Residual risks / constraints (accepted)

1. **IPC helpers** `remote:inject` / `remote:submit` / `remote:approve` are main-process test/ops surfaces. They still pass Gateway + Permission, but a compromised renderer could submit remote-shaped tasks when remote is enabled. Mitigation: keep `remote.enabled=false` until needed; desktop permission UI still gates L2/L3.
2. **Executor preference** is SkillRuntime then AgentRunService. A READ remote task may complete via `mogu.memory` without a full Brain loop unless routing/settings send it there. Not a Trust bypass; document as product behavior.
3. **QQ / WeChat** adapters are simulate-first transports. Do not enable in production defaults.
4. **Live Telegram** requires owner bot token + running desktop with confirm UI for L2/L3. Offline acceptance does not replace a 7-day soak.
5. **TaskStore `source`** remains an existing allowlist value (`unknown`) with `kind: "remote"` — intentional schema non-expansion.

## Mechanical checks performed

- Grep of `src/main/remote` for `child_process`, skill handler requires, Brain imports: none in adapters.
- Acceptance `npm run acceptance:v2.3` covers deny-without-approval path via admin YES gate and cancel path.
- Unit tests assert adapters lack `invoke` / `submitTask`.

## Verdict

Remote Workspace meets the charter hard constraints for a Default-Off Task Source.  
**Do not Default-On.** Proceed only with Telegram personal opt-in + approval + 7-day stability soak before any 2.4 autonomy work.
