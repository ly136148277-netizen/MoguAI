# MOGU 2.2 Neural Layer — Execution Contract

```yaml
status: SCAFFOLD
branch: capability/2.2
baseline: capability/2.1
feature_defaults: OFF
promotion: owner decision after protocol evidence
```

## Objective

MOGU 2.2 adds a complete Neural Layer for one bounded task at a time. It improves how an accepted
task is classified, routed, planned, supplied with context, executed through tools, traced, and
verified without changing 2.1 behavior when every 2.2 flag is off.

The single-task flow is:

```text
accepted task
  → task classification
  → model route and budget
  → bounded plan
  → context allocation
  → tool-chain execution
  → decision trace
  → verify / bounded repair / terminal outcome
```

## In scope

1. **Task classification** — assign one declared task class, confidence, constraints, and risk
   metadata. Unknown or ambiguous work remains explicit.
2. **Model routing and budget** — choose only from configured model profiles under task policy and
   budget. `allowModelFallback` defaults to `false`; fallback requires explicit user opt-in.
3. **Planner** — create and revise a finite plan for the current task. Plans do not become a global
   task graph.
4. **Context budget** — allocate bounded input context among instructions, repository evidence,
   tool results, and repair evidence with deterministic truncation metadata.
5. **Tool-chain** — select and sequence existing permission-gated 2.1 capabilities. The Neural
   Layer cannot bypass PermissionProxy, task ownership, path boundaries, or audit events.
6. **Decision trace** — emit secret-free, append-only routing, planning, context, tool, verification,
   and stop-reason records suitable for A/B attribution.
7. **Bounded closed loop** — verify the current task and permit only configured repair iterations,
   steps, tool calls, wall time, tokens, and cost. Exhaustion returns a typed terminal outcome.

## Explicit 2.3 boundary

The following are not 2.2 work:

- full DAG execution or arbitrary dependency graphs
- cross-skill self-healing or autonomous strategy switching across tasks
- global scheduling, global concurrency, or cross-task budget allocation
- task migration between workers, machines, runtimes, or sessions
- writable parallel agents or autonomous multi-task orchestration

These capabilities belong to MOGU 2.3. A 2.2 planner may describe ordered steps only inside one
accepted task and one bounded run.

## Compatibility and safety invariants

- All seven 2.2 flags default to `false` and can be evaluated independently.
- With all 2.2 flags off, the 2.1 route, tool behavior, errors, event semantics, and outputs remain
  unchanged.
- The 2.1 sealed holdout remains frozen and unopened. 2.2 development must not rewrite, inspect
  outcomes from, or tune against it.
- Model profiles contain routing metadata only. API keys, tokens, credentials, authorization
  headers, and equivalent secret fields never persist in `settings.json`.
- No model fallback is implicit. Missing configuration, capacity, credentials, or budget returns a
  typed `BLOCKED` or budget outcome.
- Every side effect still passes the existing permission and audit planes. A plan or repair
  iteration grants no additional authority.

## Feature flags and configuration

All flags are default-off:

- `v22NeuralLayer`
- `v22ModelRouting`
- `v22Planner`
- `v22ContextBudget`
- `v22ToolChain`
- `v22DecisionTrace`
- `v22ClosedLoop`

`v22Config` has four sanitized sections: `modelProfiles`, `taskPolicies`, `budget`, and
`allowModelFallback`. The last value defaults to `false` and preserves only an exact boolean `true`
user opt-in. Profiles refer to SecretStore identifiers when credentials are needed; they never
contain credential values.

## Done conditions

2.2 is complete only when all seven modules and their security tests exist, the default-off path
passes the full 2.1 regression suite, protocol comparisons pass without safety regression, decision
traces make route and budget decisions attributable, and owner review approves any promotion.
Implementation completion alone never turns a flag on.
